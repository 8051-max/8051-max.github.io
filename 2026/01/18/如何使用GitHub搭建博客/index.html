如何使用GitHub搭建博客

前言

建立一个独立博客和公众号的想法，最初源于科协学长和依门的指引。在2026年1月17日这个夜晚，我终于付诸行动，耗费数小时完成了从0到1的初建。这第一篇文章，既是对搭建全过程的回溯，也是对我所遇困难与解决思路的总结。以此为起点，开启我的记录之旅。

一、对 GitHub 的认识、 搭建博客中 GitHub 的作用与Hexo的认识

1、什么是GitHub

什么是GitHub呢？在写这篇博客之前我也不清楚，很多开发者只是把它当成一个网盘。在询问 AI ，我得到的理解是：GitHub 是一个基于 Git 的云端代码托管平台和开发者社区。在形式上，目前它主要是一个网站（GitHub.com），也提供桌面客户端和移动 App 。

如果说百度网盘是用来存电影和文档的云端仓库，那么 GitHub 就是专门用来存代码和工程项目的云端保险柜。但它不仅仅是仓库，它更像是一个程序员的社交媒体。在这里，全世界的开发者把自己的代码开源，其他人可以学习、点赞或者帮忙修改。

2、搭建博客中 GitHub 的作用

在 Hexo 搭建的过程中，GitHub 主要有两个作用：

代码仓库：它存着那些执行 hexo d 后推上来的网页文件。

网站服务器：这是 GitHub 针对开发者提供的特殊服务。它不仅帮助开发者存网页，还负责把这些文件跑起来，变成一个互联网可访问的网址。

3、什么是Hexo

Hexo 是一个基于 Node.js 的静态网站生成器 (SSG)。

静态网站:在访问时不需要查询数据库,所有页面在部署前就已经全部生成好了，因此它被访问速度极快，安全性也更高。

生成器：它像是一个加工厂。我负责提供原始的素材（Markdown 文章），它负责按照我选定的模版，把这些素材加工成网页

Hexo 的主要工作是自动化地完成 “从文本文档到完整网站” 的转换过程。

二、GitHub 账号找回与2FA 认证

为了使用 VS Code 内置的 Copilot，我此前申请过 GitHub 账号并激活了学生认证，这也是我此次选择 GitHub Pages 搭建博客的初衷。然而，由于近期较少使用 VS Code，GitHub 账号因长期未登录而略显生疏。于是，我遇到的第一个挑战并非复杂的配置命令，而是账号的找回与权限重激活。

尽管我依然记得登录密码，但由于手机端并未安装 Authenticator 身份验证器，无法通过双重身份验证（2FA）进入账户。通过翻阅邮件记录，我确认该账户已于去年 10 月 1 日开启了 2FA 保护。

最终，我利用创建账号时自动下载并保存在电脑本地的离线恢复代码（Recovery Codes）成功登录。进入账户后，我立即在移动端补齐了 Authenticator 的配置，为后续 GitHub Pages 的部署及 Hexo 环境的搭建扫清了身份验证障碍。

三、命令提示符（CMD）与基础环境配置

1、认识命令提示符（CMD）

1.1 什么是 CMD？

我们小时候对“黑客”的初印象——在幽暗的房间里对着黑色屏幕飞速敲击字符，那个黑色窗口大部分时候确实就是命令行界面（CLI），在 Windows 上就是 CMD 或 PowerShell。

CMD是 Windows 系统自带的命令行解释器。它的作用是让我们绕过图形化界面（GUI），直接通过键盘输入特定指令来调用计算机底层的功能。

1.2 什么是图形化界面（GUI）和命令行界面（CLI）？

为了不被术语绕晕，我们可以通过“表象”与“本质”的映射，快速看透计算机交互的逻辑：

图形化界面 (GUI)：是操作系统为了降低操作门槛，在复杂的底层指令上构建的一层视觉交互界面。它通过图标和窗口，将原本抽象的代码指令具象化。例如：当我们用鼠标点击桌面上的“回收站”图标时，实际上是触发了系统去读取并展示特定废弃路径文件的指令。

命令行界面 (CLI)：是计算机的原生交互方式。它跳过了所有视觉包装，让开发者通过键盘输入精确的文本指令，直接与系统内核对话。例如：当我们在终端输入ipconfig并回车时，计算机便会绕过任何菜单界面，直接从系统底层调取并返回当前的网络配置信息。

本质上，GUI 是 CLI 的实时指令翻译器：我们在屏幕上每挥动一次鼠标、每点击一个图标，实际上都等同于在后台触发并执行了一次特定的命令行指令。可以说，图形化界面只是把原本枯燥的代码逻辑，封装成了可以通过鼠标动作触发的预设程序。

我们可以通过一个最基础的操作来理解这种等价性：

GUI 动作：我们在窗口中，对着名为 hello world 的文件夹双击左键。

CMD 命令：我们在命令行中输入 cd "hello world"。

1.3 VS Code、IDE 与 CMD的对比

CMD (命令提示符)： 它是一个纯粹的命令行解释（Shell）。当使用者输入一行命令后，它传达给操作系统内核执行。它没有精美的界面，一切沟通全靠字符。

VS Code (Visual Studio Code)：它本质上是一个轻量级的代码编辑器。它的核心任务是让“写代码”写得更舒服（高亮、补全、查错）。

IDE（集成开发环境）： 它是工具的集合体。IDE 将代码编写、编译、调试等全流程功能深度整合，旨在提供一套完整且标准的开发环境，极大降低了环境搭建的门槛

有两点非常有意思：VS Code 通过安装插件，变成一个极其强大的 IDE；VS Code 底部的终端实际上就是电脑自带的CMD

1.4 为什么在搭建博客时反复使用CMD？

工具链调用：我们安装的 Node.js、Git 以及 Hexo，本质上都没有“桌面图标”，它们需要通过在 CMD 中输入特定的命令来唤醒。

确定性反馈：每一个操作（如生成网页、上传代码）执行后，CMD 会实时返回运行日志。如果执行失败，它会给出具体的错误代码，方便我们定位问题。

1.5如何进入博客工作区？

按 Win + R 键，输入 cmd 并回车。

2、基础环境配置

2.1 Git 、​Node.js 与 Hexo 的安装与检验

通过 Gemini 我了解到写博客之前需要安装两个软件：

Git：代码运输车。它的作用是把代码在“本地”和“GitHub 仓库”之间运输。

安装链接：Git for Windows：https://git-scm.com/download/win

下载后安装时，一直点击 “Next” 即可，默认配置已经能完美兼容 Hexo 的推送需求。

​Node.js：运行引擎。Hexo 是用 JavaScript 写的，Windows 本身看不懂 JS，必须装个 Node.js 才能把 Hexo 跑起来。

安装链接：Node.js (LTS 版本)：https://nodejs.org/

下载左侧标有 LTS（长期支持版）的安装包，它在处理 Hexo 依赖时最稳定。

在下载之前我们需要检查电脑上是否曾经安装过这两个软件，我们首先打开命令CMD，然后分别输入node -v 与 git --version（注意二者中间都有空格，后面的命令可以写作git -v ，但是兼容性不如第一个）再回车，如果出现版本号就说明已下载，如果没有则点击链接分别下载即可。

安装完成后我们需要使用 CMD 安装 Hexo :打开 CMD 输入

npm install -g hexo-cli

  1. npm：Node Package Manager的缩写，这是 Node.js 的包管理器，可以把它类比成“软件商店”或“插件仓库”。它负责从网上下载并管理各种基于 JavaScript 的工具和程序
  2. install：这是向 npm 下达的具体动作命令，意思是安装。它告诉电脑要去 npm 的官方服务器上寻找指定的安装包并下载到本地。
  3. -g (Global)：这是一个“参数标志”，代表“全局安装”。
    如果没有这个 -g，软件只会安装在当前的文件夹里。
    加上 -g 后，这个工具会被安装到系统的全局路径中。这意味着以后无论在哪个文件夹打开命令提示符，都可以直接调用 hexo 命令。
  4. hexo-cli：这是要安装的软件名称。cli 代表 Command Line Interface（命令行界面）。它并不是博客本身，而是管理博客的一套工具箱。只有装了它，电脑才能听懂 hexo init、hexo g 这些指令。

当我们输入这行代码并回车后，npm 会联网下载 Hexo 的核心工具包。安装完成后，我们的系统里就多了一个名为 hexo 的新指令。（npm 之于 Node.js，就像 pip 之于 Python）

2.2 Hexo的工程初始化与依赖装配

在 CMD 中输入 d: 回车（切换到D盘）（这一步可有可无，我没有做）

输入hexo init blog

  1. ​hexo (主语)：
    ​呼叫主程序。相当于在 CMD 里大喊一声“Hexo！”，唤醒刚才安装好的那个 CLI 工具。

  2. ​init (谓语)：
    ​全称 Initialize（初始化）。这是计算机领域的标准术语，意思是从零开始建立基础结构。
    ​它的作用是从 GitHub 上把 Hexo 的官方脚手架（Scaffolding） 克隆下来。

  3. ​blog (宾语)：
    ​目标文件夹的名字。这告诉 Hexo：“请给我新建一个叫 blog 的文件夹，把所有的东西都装进去。”
    ​注意：这个名字是我们自己定的。如果我们写 hexo init website，它就会新建一个叫 website 的文件夹，同时 Hexo 的程序会自动去 GitHub 上把一个标准模版搬运下来。这个模版里自带 package.json。

(这一步会从网上下载很多文件，可能需要等一两分钟)

输入 cd blog 进入文件夹，然后输入npm install将所有依赖组件全部下载（我们已经运行了 hexo init blog，并进入了blog文件夹。这时候文件夹里虽然没有写博客，但里面已经有了 package.json 文件。程序会读取package.json 文件中列出的所有依赖组件，并将他们批量下载到本地。

然后输入 hexo s，如果看到屏幕上出现 Hexo is running at http://localhost:4000，说明成功了！

（​这行代码是 Hexo Server（Hexo 服务器）的缩写。
它会在我们的本地电脑上启动一个临时的微型网站服务器。
作用是本地预览 Local Preview。在不联网、不上传的情况下，在自己的浏览器里先看一眼效果。而且只要关闭了 CMD ，网站便打不开了）

四、打通云端通道：配置 Deployer 与正式发布

1、让本地服务器停下来：

在 CMD 中按 Ctrl+C ,直到出现……\blog>。

2、安装“快递员”插件

Hexo 需要一个专门的插件把文章“寄”给 GitHub。我们需要输入这个命令：npm install hexo-deployer-git --save

  1. ​npm install：​动作：去下载。

  2. ​hexo-deployer-git：​对象：这是插件的名字。
    ​含义：Hexo 的 Deployer (部署器)，专门用于 Git 协议。
    ​注意：Hexo 核心程序是很精简的，它默认并不包含“推送到 GitHub”这个功能，所以必须额外安装这个插件。

  3. ​--save：
    ​参数：“写入清单”。
    ​作用：它告诉 npm：“不仅要帮我下载这个插件，还要把它的名字自动写进 package.json。”
    ​这样做的用处是：以后如果我们换了电脑，只需要运行一次 npm install，系统就会根据清单自动把它装回来，不用再记一遍插件名。

​为什么要执行这一步？

​我们之前运行 hexo s 只是在本地看。
接下来的目标是运行 hexo d (deploy)，把博客发到互联网上。这些就是准备工作.

3、修改配置文件

这是最细致的一步，请务必小心。

  1. 打开电脑文件夹，找到我们刚才建博客的地方（……\blog）。
  2. 找到一个叫 _config.yml 的文件（注意：不是 _config.landscape.yml，是短的那个）。
  3. 右键 -> 打开方式 -> 记事本（如果有 VS Code 更好，没有就用记事本）。
  4. 拉到文件的最最最底下，我们会看到一个 deploy:。
  5. 把最后那几行全部删掉，替换成下面这段代码

type: git：
​含义：运输方式。
​解释：告诉 Hexo，使用 Git 协议来发货。这就激活了我们刚才安装的那个 hexo-deployer-git 插件。

​repo: ...：
​含义：收货地址。
​解释：git@github.com:github账号名称/github账号名称.github.io.git 是我们 GitHub 仓库的 SSH 地址。这告诉插件：“把生成的网页文件，全部推送到 GitHub 上名为github账号名称的那个仓库里去。”

​branch: main：
​含义：货架/分支。
​解释：告诉 GitHub，“请把这些文件放在 main 这个主分支上展示。”

注意：type: 前面要有 2个空格 的缩进，冒号后面也要有 1个空格。直接复制上面这段最稳妥。
保存文件，然后关闭记事本。

4、 呼叫主程序并部署到网页

输入 hexo d

当敲下回车后，Hexo 会指挥那个“快递员插件”

（hexo-deployer-git）执行以下一系列复杂的连招：

​打包：它会找到 public 文件夹里所有刚刚生成好的 HTML 网页。

​建立连接：它读取 _config.yml 里的地址，通过 SSH 协议连接到 GitHub。

​推送 (Push)：它执行类似 git push 的操作，把这些文件强行推送到 GitHub 仓库的 main 分支。

然后博客就上传成功了

在之后写博客时，我们常用hexo cl && hexo g -d

(先清理缓存 Clean，再生成 Generate，最后部署 Deploy。这是最干净、最不容易报错的流程。)

五、实际部署时遇到的两个报错

在配置完 _config.yml 后，我运行 hexo d 进行部署，期间遇到了两个报错。这里记录一下排查和解决的过程。

报错一：Permission denied (权限被拒绝)

第一次执行部署命令时，终端报错：git@github.com: Permission denied (publickey)。

原因： 这是因为我的电脑（新环境）没有配置 SSH 密钥，GitHub 无法验证我的身份，所以拒绝了连接。

解决步骤：

在 CMD 输入生成密钥的指令

ssh-keygen -t ed25519 -C "你的邮箱地址"

生成完毕后，查看公钥内容：

type %userprofile%\.ssh\id_ed25519.pub

复制显示出来的乱码字符串，登录 GitHub，进入 Settings -> SSH and GPG keys，点击 New SSH key，将内容粘贴进去并保存。

报错二：Repository not found (找不到仓库)

配置好 SSH Key 后，我再次尝试部署，结果报错：ERROR: Repository not found。

原因： 这是一个逻辑遗漏。虽然我在 _config.yml 配置文件里填好了仓库地址，但我并没有在 GitHub 上实际创建这个仓库。

解决步骤：

登录 GitHub，点击右上角 + 号，选择 New repository。

关键点：仓库名必须严格填写为 GitHub名字.github.io。如果格式不对，后续将无法生成正确的博客网址。

点击创建。

部署成功
修正了上述两个问题后，我第三次输入 hexo d。 这次终端显示绿色提示 INFO Deploy done: git。随后我访问了博客网址，页面已经可以正常打开。
